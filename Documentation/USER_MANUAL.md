## 第一部分：平台概述

### 1.1 平台定位

SAnalysis 是一套**六队协同的妖股（Meme Stock）分析系统**。它模拟了一支由六个专业小队组成的侦察部队，每支小队从不同维度对股票进行量化评分，最终通过加权融合、共振检测生成统一的观察名单（Watchlist）。

**平台核心使命**：发现能实现 **10 倍以上回报** 的爆发性股票。大盘股不可能在合理时间框架内翻 10 倍，因此平台通过全局市值门控（$5B 上限）和团队级市值门控，专注于小盘/微盘股。

平台专注于捕捉以下类型的交易机会：

| 机会类型 | 负责团队 | 核心逻辑 |
|---------|---------|---------|
| 空头挤压（Short Squeeze） | 🔴 红队 | 高做空比例 + 回补困难 + 催化剂 |
| Gamma 挤压（Gamma Squeeze） | 🟠 橙队 | 期权做市商负 Gamma 敞口 + 异常期权流 |
| 社交情绪爆发 | 🟡 黄队 | Reddit/ApeWisdom 提及飙升 + 看涨情绪 |
| 低流通股突破 | 🟢 绿队 | 小流通盘 + 量能爆发 + 技术突破形态 |
| 动量催化融合 | 🔵 蓝队 | 多时间框架动量 + 财报催化 + 市场环境 |
| **10 倍结构潜力** | 🟣 紫队 | **市值层级 + 流通结构 + 稀释风险 + 爆发蓄势** |

### 1.2 设计理念

1. **多维度交叉验证**：六支团队独立评估，当 3+ 支团队同时命中同一标的时触发**共振加成**（×1.25），置信度显著提升
2. **10x 专注**：全局市值门控（$5B）+ 团队级市值门控（$2B-$5B），排除不可能翻 10 倍的大盘股
3. **配置驱动**：所有阈值均可通过 YAML 文件调整，无需改代码
4. **零成本数据源**：Phase 1 仅使用免费 API（yfinance、Finviz 免费版、ApeWisdom、Google Trends）
5. **缓存优先**：文件级缓存避免重复 API 调用，默认 TTL 4 小时
6. **并行执行**：六支团队通过线程池并行运行，缩短扫描时间
7. **AI 增强**（v0.5.0）：可选的 Claude AI 定性重评分层，捕捉公式无法识别的细微风险（如拉高出货、虚假突破、回音室效应）

### 1.3 整体架构

```
数据层 (src/utils/)
  market_data.py   ── yfinance 封装（价格/基本面/期权/财报/VIX）
  finviz_scraper.py ── Finviz 免费版 HTML 爬虫（筛选器）
  technical.py     ── 技术指标库（RSI/MACD/BB/OBV/VWAP/ATR/SMA/EMA）

核心层 (src/core/)
  config.py   ── 线程安全的单例配置（YAML + 环境变量）
  cache.py    ── 文件缓存（Parquet + JSON，原子写入）
  base.py     ── BaseScreener 抽象基类（fetch_candidates + analyze 契约）
  data_types.py ── 标准数据结构（ScreenResult, ShortData, OptionsSnapshot 等）
  exceptions.py ── 自定义异常层次（SAnalysisError, AIRescoreError 等）

AI 层 (src/ai/)                    ← v0.5.0 新增
  client.py   ── Anthropic SDK 封装（线程安全单例，重试，优雅降级）
  prompts.py  ── 五支团队专用系统提示词 + 消息构建器
  data_types.py ── AIRescoreResult 数据结构

团队层 (src/teams/)
  red/screener.py    ── 空头挤压狙击队
  orange/screener.py ── Gamma 挤压猎手
  yellow/screener.py ── 社交情绪侦察队
  green/screener.py  ── 低流通突破特战队
  blue/screener.py   ── 动量催化复合精英队
  purple/screener.py ── 十倍潜力评估师 (v1.0.0 新增)

管线层 (src/pipeline/)
  orchestrator.py ── 多团队并行调度 + AI 重评分 + 共振检测 + 加权综合评分 + CSV 输出

入口层
  main.py ── CLI 命令行入口
```

### 1.4 数据流

```
[Finviz/yfinance/ApeWisdom/Reddit/Google Trends]
            |
            v
    候选股发现 (fetch_candidates)
    -- 各团队独立获取候选列表
            |
            v
    单股深度分析 (analyze)
    -- 4 因子评分，每因子 0-25 分，总计 0-100 分
    -- 输出 ScreenResult 标准数据结构
            |
            v
    管线编排 (PipelineOrchestrator)
    -- 按 ticker 合并各团队结果
    -- [可选] AI 智能重评分 (--ai-rescore)
       调用 Claude API 对每队结果定性评估
       混合评分: blended = quant × 0.7 + ai × 0.3
    -- 计算加权综合评分 (composite_score)
       权重: 红×1.2, 橙×1.0, 黄×0.7, 绿×1.2, 蓝×0.5, 紫×1.3
    -- 共振检测: 3+队命中同一标的 → 综合分 ×1.25
    -- 全局市值门控: > $5B 的标的被最终过滤
    -- 统计命中团队数 (team_count)
            |
            v
    输出 (CSV + 控制台摘要)
    -- data/output/watchlist_YYYYMMDD_HHMMSS.csv
```

## 第二部分：快速上手

### 2.1 环境要求

| 项目 | 要求 |
|------|------|
| Python | 3.12+ |
| 操作系统 | Linux (Ubuntu 24.04 / WSL2 已验证)、macOS |
| 网络 | 需要访问 Yahoo Finance、Finviz、ApeWisdom、Google Trends |
| 内存 | 建议 2GB+ |
| 磁盘 | 缓存约占 50-200MB |

### 2.2 安装步骤

```bash
# 1. 克隆仓库
git clone <repo-url> SAnalysis
cd SAnalysis

# 2. 创建 Python 虚拟环境
python3.12 -m venv .venv
source .venv/bin/activate

# 3. 安装依赖
pip install -r requirements.txt
# 或从 pyproject.toml 安装
pip install -e .

# 4. (可选) 安装 Reddit API 支持
pip install -e ".[reddit]"

# 5. (可选) 安装 AI 重评分支持
pip install -e ".[ai]"

# 6. (可选) 配置 Reddit API 密钥
cp config/secrets.yaml.example config/secrets.yaml
# 编辑 secrets.yaml，填入 Reddit API 凭据
```

**核心依赖清单：**

| 包名 | 用途 |
|------|------|
| yfinance | 股价、基本面、期权、财报、VIX |
| pandas / numpy | 数据处理 |
| pandas_ta | 技术指标（有手动回退方案） |
| pyyaml | 配置文件解析 |
| requests | HTTP 请求 |
| beautifulsoup4 + lxml | Finviz HTML 解析 |
| vaderSentiment | 情绪分析 |
| pytrends | Google Trends 数据 |
| plotly / matplotlib | 可视化（预留） |
| pyarrow | Parquet 缓存序列化 |
| praw (可选) | Reddit API 直接扫描 |
| anthropic (可选) | Claude AI 重评分 |

### 2.3 首次运行

```bash
# 激活虚拟环境
source .venv/bin/activate

# 运行全部 6 支团队（默认模式）
python main.py
```

首次运行将需要较长时间（3-10 分钟），因为需要从多个 API 获取并缓存数据。后续运行在缓存有效期内（默认 4 小时）会显著加速。

### 2.4 CLI 参数详解

```bash
python main.py [选项]
```

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `--version`, `-V` | 标志 | -- | 显示版本号并退出 |
| `--teams` | 列表 | 全部6队 | 指定运行哪些团队。可选值：`red`, `orange`, `yellow`, `green`, `blue`, `purple` |
| `--tickers` | 列表 | 各团队自行发现 | 指定分析的股票代码，跳过候选发现阶段 |
| `--top` | 整数 | 20 | 控制台输出显示前 N 名结果 |
| `--no-parallel` | 标志 | False | 串行模式运行（用于调试） |
| `--no-save` | 标志 | False | 不保存 CSV 文件 |
| `--clear-cache` | 标志 | False | 运行前清除所有缓存数据 |
| `--ai-rescore` | 标志 | False | 启用 Claude AI 定性重评分（需 `ANTHROPIC_API_KEY`） |
| `--log-level` | 选择 | INFO | 日志级别：`DEBUG`, `INFO`, `WARNING`, `ERROR` |

**常用命令示例：**

```bash
# 只跑红队和绿队
python main.py --teams red green

# 对指定股票进行全团队分析
python main.py --tickers GME AMC TSLA

# 调试模式：串行运行 + 详细日志
python main.py --no-parallel --log-level DEBUG

# 清缓存后重新扫描，显示 Top 30
python main.py --clear-cache --top 30

# 组合使用：只用红队分析特定股票
python main.py --teams red --tickers GME AMC CLOV

# 快速扫描，不保存文件
python main.py --no-save --top 10

# 启用 AI 重评分（全部团队）
ANTHROPIC_API_KEY=sk-ant-xxx python main.py --ai-rescore

# AI 重评分 + 指定团队和标的
ANTHROPIC_API_KEY=sk-ant-xxx python main.py --ai-rescore --teams red green --tickers GME AMC
```

## 第三部分：六支团队详解

---

### 3.1 红队 -- 空头挤压狙击队 (ShortSqueezeScreener)

**文件**: `src/teams/red/screener.py`
**配置键**: `red_team`

#### 3.1.1 使命与定位

红队专注于发现**空头挤压（Short Squeeze）**潜力最高的标的。空头挤压是指当高比例做空的股票价格开始上涨时，做空者被迫买入回补仓位，进一步推升价格的正反馈循环。

红队寻找的理想标的特征：高做空浮动比（SI% of Float）、高借券成本、低流通盘、即将到来的催化剂（如财报）、技术面看多。

#### 3.1.2 候选股筛选逻辑 (fetch_candidates)

**主数据源**: Finviz 免费版筛选器

默认筛选条件（可通过 `red_team.finviz_filters` 配置）：

| 筛选项 | 默认值 | 含义 |
|--------|--------|------|
| `short_float` | `over10` | 做空浮动比 > 10% |
| `average_volume` | `over200k` | 日均成交量 > 20 万股 |
| `price` | `over1` | 股价 > $1（排除仙股） |

**回退候选列表**：当 Finviz 请求失败或返回空结果（可能触发频率限制）时，使用维护的候选池：GME, AMC, CLOV, SPCE, WKHS, GOEV, RKT, PLTR, BB, NOK, FUBO, MVIS, SNDL, TLRY, SOFI, MARA, RIOT。

**准入门控**（在 `analyze()` 中执行）：

| 门控条件 | 配置参数 | 默认值 | 说明 |
|----------|----------|--------|------|
| 最低做空浮动比 | `min_short_float_pct` | 10.0% | SI% 低于此值直接跳过 |
| 最低日回补天数 | `min_days_to_cover` | 3.0 天 | DTC 低于此值直接跳过 |
| 最大市值上限 | `max_market_cap_millions` | 2000（$2B） | 市值超过此值直接跳过（v1.0.0 从 $10B 收紧至 $2B） |

**数据验证**：SI% > 100% 的异常数据（常见于 ADR）会被截断至 100% 并记录警告日志。

#### 3.1.3 四因子评分模型 (总分 0-100)

##### 因子一：做空强度 (short_intensity) -- 0~25 分

**评分依据**：做空浮动比（Short Interest % of Float）

红队使用**分段线性插值（Piecewise Linear Interpolation）**替代阶梯函数，消除评分断崖效应（例如 SI=9.9% 和 SI=10.0% 之间不再有突然的分数跳跃）。

**插值断点表：**

| SI% 范围 | 分数范围 | 含义 |
|----------|----------|------|
| 0% | 0.0 分 | 无做空信号 |
| 0% ~ 10% | 0.0 ~ 10.0 分（线性） | 低做空水平 |
| 10% ~ 15% | 10.0 ~ 14.0 分（线性） | 中等做空 |
| 15% ~ 20% | 14.0 ~ 18.0 分（线性） | 高做空 |
| 20% ~ 30% | 18.0 ~ 22.0 分（线性） | 非常高 |
| 30% ~ 40% | 22.0 ~ 25.0 分（线性） | 极端做空 |
| >= 40% | 25.0 分（满分） | 极端挤压潜力 |

**示例**：SI = 12.5% 时，位于 10%-15% 区间，分数 = 10.0 + (14.0 - 10.0) x (12.5 - 10.0) / (15.0 - 10.0) = 12.0 分。

##### 因子二：回补难度 (cover_difficulty) -- 0~25 分

**评分依据**：空头平仓的难度，由三个子指标加总（上限 25 分）。

**子指标 A -- 日回补天数 (Days to Cover, DTC)：0~12 分**

| DTC 范围 | 分数 | 含义 |
|----------|------|------|
| >= 10 天 | 12.0 | 空头几乎无法快速退出 |
| >= 7 天 | 10.0 | 严重拥挤 |
| >= 5 天 | 8.0 | 高度拥挤 |
| >= 3 天 | 5.0 | 中度拥挤 |
| < 3 天 | DTC x 1.5 | 低拥挤 |

**子指标 B -- 流通盘紧度 (Float Tightness)：0~8 分**

| 流通盘（百万股） | 分数 | 含义 |
|----------------|------|------|
| < 10M | 8.0 | 极小流通盘，极易挤压 |
| < 20M | 6.0 | 小流通盘 |
| < 50M | 4.0 | 中小流通盘 |
| < 100M | 2.0 | 中等流通盘 |
| >= 100M | 0.0 | 大流通盘，不利于挤压 |

**子指标 C -- 认沽/认购比 (Put/Call Ratio, PCR)：0~5 分**

要求最低期权总成交量 >= 100 手（`_MIN_PCR_VOLUME = 100`），低于此阈值视为数据不可靠，不参与评分。

| PCR 范围 | 分数 | 含义 |
|----------|------|------|
| > 2.0 | 5.0 | 极端看空仓位，大量挤压燃料 |
| > 1.5 | 3.0 | 偏空仓位 |
| > 1.0 | 1.5 | 轻微偏空 |
| <= 1.0 | 0.0 | 无看空信号 |

##### 因子三：催化剂临近 (catalyst_proximity) -- 0~25 分

**评分依据**：两个子指标加总（上限 25 分）。

**子指标 A -- 财报临近度：0~12 分**

| 距下次财报天数 | 分数 | 含义 |
|--------------|------|------|
| <= 7 天 | 12.0 | 财报窗口期，高度关注 |
| <= 14 天 | 8.0 | 进入预热期 |
| <= 30 天 | 4.0 | 早期预期 |
| > 30 天 | 0.0 | 无催化剂 |

**子指标 B -- 相对成交量 (RVOL, 30 日均量对比)：0~10 分**

| RVOL | 分数 | 含义 |
|------|------|------|
| >= 5.0x | 10.0 | 极端活跃 |
| >= 3.0x | 7.0 | 高度活跃 |
| >= 2.0x | 4.0 | 活跃 |
| >= 1.5x | 2.0 | 轻度活跃 |
| < 1.5x | 0.0 | 正常 |

##### 因子四：技术动量 (technical_momentum) -- 0~25 分

**评分依据**：四个技术子指标加总（上限 25 分），使用 3 个月历史数据。

**子指标 A -- RSI(14) 定位：0~8 分**

| RSI 范围 | 分数 | 含义 |
|----------|------|------|
| 50 ~ 70 | 8.0 | 健康多头动量，尚有上涨空间 |

#### 3.1.5 特殊机制

**数据过期预警**：红队会检查做空数据的报告日期（`dateShortInterest`）。FINRA 约每 15 天发布一次做空数据，当数据超过 16 天（`_MAX_SI_AGE_DAYS = 16`）时，系统会输出 WARNING 日志提醒用户数据可能已过时。

**缺失数据处理**：当 `days_to_cover` 或 `put_call_ratio` 数据不可用时，信号值设为 `NaN`（而非 0），方便下游区分"数据缺失"和"值为零"。

#### 3.1.6 信号解读

- SI 数据延迟：FINRA 报告有 2 周延迟，实际 SI 可能已大幅下降
- 大市值股票（> $10B）默认被门控排除，因为做空者资金量大，更不易被挤压

### 3.2 橙队 -- Gamma 挤压猎手 (GammaSqueezeScreener)

**文件**: `src/teams/orange/screener.py`
**配置键**: `orange_team`

#### 3.2.1 使命与定位

橙队专注于发现**Gamma 挤压（Gamma Squeeze）**潜力标的。Gamma 挤压是指当做市商因持有大量空头看涨期权头寸而处于"负 Gamma"状态时，股价上涨迫使他们买入正股对冲，形成正反馈上涨循环。

橙队分析期权市场的三个关键维度：异常期权活动、Gamma 敞口分布、隐含波动率结构。

#### 3.2.2 候选股筛选逻辑 (fetch_candidates)

Phase 1 使用维护的高流动性可期权交易标的列表：

```
GME, AMC, TSLA, NVDA, AAPL, PLTR, SOFI, NIO, RIVN, LCID,
CLOV, FUBO, MARA, RIOT, COIN, HOOD, SNAP, PINS, RBLX,
DKNG, SPCE, UPST, SMCI, ARM
```

**数据获取**：通过 yfinance 获取最近到期日（默认 30 天内）的期权链数据，包括看涨期权（calls）和看跌期权（puts）的行权价、成交量、持仓量、隐含波动率和希腊值。

#### 3.2.3 四因子评分模型 (总分 0-100)

##### 因子一：期权活跃度 (options_activity) -- 0~25 分

**前置门控**：总期权成交量（看涨 + 看跌）必须 >= `min_option_volume`（默认 1000 手），否则直接返回 0 分。

**子指标 A -- 成交量水平：0~8 分**

| 总期权成交量 | 分数 | 含义 |
|------------|------|------|
| >= 50,000 | 8.0 | 机构级活跃 |
| >= 20,000 | 6.0 | 高度活跃 |
| >= 5,000 | 4.0 | 活跃 |
| >= 1,000 | 2.0 | 基本活跃 |

**子指标 B -- 异常活动（Vol/OI 比）：0~10 分**

当看涨期权出现成交量远超持仓量的情况，说明有大量新仓位正在激进开仓。

| Vol/OI 最大比值 | 分数 | 含义 |
|----------------|------|------|
| >= 10x | 10.0 | 极端异常，大规模新仓位 |
| >= 5x | 7.0 | 显著异常 |
| >= 3x | 4.0 | 轻度异常 |
| < 3x | 0.0 | 正常 |

阈值 `unusual_volume_ratio` 可配置，默认 3.0。

**子指标 C -- OTM 看涨期权集中度：0~7 分**

价外看涨期权（strike > spot）成交量占总看涨成交量的比例。高 OTM 集中度意味着市场在押注大幅上涨。

| OTM 看涨集中度 | 分数 | 含义 |
|---------------|------|------|
| > 70% | 7.0 | 极端看涨押注 |
| > 50% | 4.0 | 偏看涨 |
| > 30% | 2.0 | 轻度看涨 |
| <= 30% | 0.0 | 正常 |

##### 因子二：Gamma 敞口 (gamma_exposure) -- 0~25 分

**核心概念**：GEX（Gamma Exposure）衡量做市商的 Gamma 敞口方向和强度。

**符号约定**：
- `net_gex = call_gex - put_gex`
- `net_gex > 0`：做市商处于负 Gamma 状态，价格变动会被**放大**，利于挤压
- `net_gex < 0`：做市商处于正 Gamma 状态，价格变动会被**抑制**

当期权链数据包含 Greeks 时使用真实 Gamma 计算（`GEX = Gamma x OI x 100 x Spot`），否则使用持仓量集中度作为代理指标。

**子指标 A -- GEX 方向与幅度归一化：0~14 分**

采用**不平衡比率**量化 GEX 偏斜程度：`imbalance = |net_gex| / (|call_gex| + |put_gex|)`

| 条件 | 分数 | 含义 |
|------|------|------|
| net_gex > 0 且 imbalance >= 0.7 | 14.0 | 极端负 Gamma，强烈放大效应 |
| net_gex > 0 且 imbalance >= 0.4 | 11.0 | 显著不平衡 |
| net_gex > 0 且 imbalance < 0.4 | 8.0 | 轻度正 GEX（有一定挤压潜力） |
| net_gex == 0 | 3.0 | 中性，对冲平衡 |
| net_gex < 0 | 1.0 | 做市商长 Gamma，抑制波动 |

**子指标 B -- 最大 Gamma 行权价临近度：0~7 分**

最大 Gamma 行权价是 Gamma 钉扎效应最强的价位，当现价接近此处时意味着突破或钉扎。

| 临近度（与 spot 的距离/spot） | 分数 | 含义 |
|-----------------------------|------|------|
| < 2% | 7.0 | 极度接近，强烈钉扎/突破潜力 |
| < 5% | 4.0 | 较近 |
| < 10% | 2.0 | 有一定影响 |
| >= 10% | 0.0 | 无显著影响 |

**子指标 C -- GEX 翻转点临近度：0~4 分**

GEX 翻转点（Flip Point）是累积净 GEX 从正变负或反之的价位，代表做市商对冲行为的切换边界。

| 翻转点距离（与 spot 的距离/spot） | 分数 | 含义 |
|---------------------------------|------|------|
| < 5% | 4.0 | 翻转点非常近，高转换风险 |
| >= 5% 但存在翻转点 | 2.0 | 翻转点存在但较远 |
| 无翻转点 | 0.0 | -- |

##### 因子三：隐含波动率动态 (iv_dynamics) -- 0~25 分

**子指标 A -- ATM 隐含波动率：0~10 分**

| ATM IV | 分数 | 含义 |
|--------|------|------|
| >= 150% | 10.0 | 市场预期剧烈波动 |
| >= 100% | 8.0 | 高波动预期 |
| >= 60% | 5.0 | 中等 |
| >= 30% | 2.0 | 正常 |
| < 30% | 0.0 | 低波动 |

**子指标 B -- IV 偏斜 (Put IV - Call IV)：0~8 分**

| IV 偏斜 | 分数 | 含义 |
|---------|------|------|
| > 0.2（看跌 IV 溢价大） | 8.0 | 大量看跌对冲需求，潜在挤压燃料 |
| > 0.1 | 5.0 | 中等对冲需求 |
| > 0 | 2.0 | 轻度偏斜 |
| < -0.1（看涨 IV 溢价大） | 7.0 | 激进看涨买入 |

##### 因子四：持仓结构 (oi_setup) -- 0~25 分

**前置门控**：总持仓量（call_OI + put_OI）必须 > 0，否则返回 0 分。

**子指标 A -- 绝对持仓量：0~7 分**

| 总 OI | 分数 | 含义 |
|-------|------|------|
| >= 500,000 | 7.0 | 巨量持仓待平仓 |
| >= 100,000 | 5.0 | 大量 |
| >= 50,000 | 3.0 | 中量 |
| < 50,000 | 1.0 | 少量 |

**子指标 B -- 看涨 OI 占比：0~6 分**

| 看涨 OI / 总 OI | 分数 | 含义 |
|-----------------|------|------|
| > 65% | 6.0 | 大量看涨持仓，做市商大量空 Gamma |
| > 55% | 3.0 | 偏看涨 |
| <= 55% | 0.0 | 平衡或偏看跌 |

**子指标 C -- OI 加权认沽/认购比 (PCR_OI)：0~5 分**

OI 加权的 PCR 比成交量 PCR 更稳定（减少盘中噪音）。

| PCR_OI | 分数 | 含义 |
|--------|------|------|
| > 2.0 | 5.0 | 极端看跌对冲（挤压燃料） |
| < 0.3 | 5.0 | 极端看涨主导（挤压预期） |
| > 1.5 | 3.0 | 偏空 |
| < 0.5 | 3.0 | 偏多 |
| 其他 | 0.0 | 中性 |

**子指标 D -- 成交量/OI 比 (新仓开仓强度)：0~4 分**

| Vol / OI | 分数 | 含义 |
|----------|------|------|
| > 0.5 | 4.0 | 大量新仓正在开立 |
| > 0.3 | 2.0 | 中等新仓 |
| <= 0.3 | 0.0 | 正常 |

#### 3.2.4 配置参数

```yaml

  min_option_volume: 1000       # 最低总期权成交量门控
  unusual_volume_ratio: 3.0     # Vol/OI 异常活动阈值
  min_iv_percentile: 50         # 最低 IV 百分位（Phase 2）
  gex_calculation_enabled: true # 是否启用 GEX 计算
  near_expiry_days: 30          # 关注的最近到期日范围
  spot_price_range_pct: 20      # GEX 计算的 spot 上下范围
  strike_step_count: 50         # GEX 计算的行权价数量
```

#### 3.2.5 特殊机制

**Greeks 不可用回退**：当 yfinance 返回的期权链不包含 Gamma 值时（某些标的或市场时段），系统自动切换为 OI 集中度代理方案（`_oi_concentration_proxy`），用持仓量分布替代 GEX 计算。

**GEX 翻转点算法**：使用逐行权价的累积净 GEX（`cumsum`），找到累积值符号变化的行权价，而非简单的逐行权价符号变化。

#### 3.2.6 信号解读

| 分数区间 | 含义 |
|----------|------|
| 70-100 | 极强 Gamma 挤压信号：异常期权流 + 负 Gamma + 高 IV + 看涨持仓主导 |
| 50-70 | 强信号：多数条件满足 |
| 30-50 | 中等：部分信号存在 |
| 10-30 | 弱：有基本期权活动但无明显挤压迹象 |
| 0-10 | 无信号 |

**常见陷阱**：
- 高 IV 不等于必然上涨：IV 高也可能预示大幅下跌（如财报前）
- Phase 1 的 GEX 计算是近似值：真正的 GEX 需要做市商持仓数据（Phase 2 将接入 SpotGamma）
- 只分析最近一个到期日：多到期日联合分析将在 Phase 2 实现

### 3.3 黄队 -- 社交情绪侦察队 (SocialSentimentScreener)

**文件**: `src/teams/yellow/screener.py`
**配置键**: `yellow_team`

#### 3.3.1 使命与定位

黄队负责量化**社交媒体情绪和关注度动量**。在妖股交易中，社交媒体（尤其是 Reddit 的 r/wallstreetbets）往往是重要的催化剂。黄队追踪的不仅是当前关注度水平，还包括关注度的加速变化率。

#### 3.3.2 候选股筛选逻辑 (fetch_candidates)

**数据源 2 -- Reddit 直接扫描**（需 praw + API 凭据）：扫描配置的 subreddit 热帖，提取股票代码提及，按频率排序取前 50。

```yaml
  sentiment_bullish_threshold: 0.3  # VADER 看涨阈值
  sentiment_bearish_threshold: -0.3 # VADER 看跌阈值
  mention_spike_multiplier: 3.0  # 提及飙升倍数阈值
```

#### 3.3.5 特殊机制

**Google Trends 熔断器**：Google Trends API 容易触发频率限制。黄队内置熔断器（Circuit Breaker）：当连续失败达到 3 次（`_GT_MAX_CONSECUTIVE_FAILURES = 3`）时，熔断器打开，剩余所有标的的 GT 查询直接跳过并返回 None。成功的缓存命中会重置失败计数器。

**Reddit 客户端缓存**：praw.Reddit 实例在一次扫描中只创建一次（`_get_reddit_client()` 懒初始化），所有标的共用同一个 OAuth 会话，避免每个标的重复建立连接。

**ApeWisdom 全量缓存**：第一次调用 `_fetch_apewisdom()` 时缓存完整的前 50 名数据（包括提及次数、排名、24 小时前提及次数），后续按标的查询直接从缓存读取。

#### 3.3.6 信号解读

| 分数区间 | 含义 |
|----------|------|
| 70-100 | 极强社交热度：多数据源确认 + 看涨情绪 + 加速增长 |
| 50-70 | 强信号：显著社交关注 + 看涨倾向 |
| 30-50 | 中等：有一定关注度 |
| 10-30 | 弱：边缘信号 |
| 0-10 | 社交沉默 |

**常见陷阱**：
- 无 Reddit API 凭据时情绪极性因子始终为 0，最高理论得分 75
- Google Trends 熔断器打开后动量因子中的 GT 部分失效
- 提及频率高但情绪为负面（如集体看跌）不应视为看涨信号
- 机器人/水军可以人为操纵提及数（`reddit_min_score`/`reddit_min_karma` 参数有助于过滤）

### 3.4 绿队 -- 低流通突破特战队 (LowFloatBreakoutScreener)

**文件**: `src/teams/green/screener.py`
**配置键**: `green_team`

#### 3.4.1 使命与定位

绿队专注于**低流通盘股票的技术突破机会**。低流通盘意味着在市场上可交易的股票数量有限，当买盘涌入时，价格变动会被极度放大。绿队寻找的是"盘整后放量突破"的经典技术形态。

#### 3.4.2 候选股筛选逻辑 (fetch_candidates)

**主数据源**：Finviz 免费版

默认筛选条件：

| 筛选项 | 默认值 | 含义 |
|--------|--------|------|
| max_float | under50m | 流通股数 < 5000 万股 |
| min_rvol | over2 | 相对成交量 > 2.0x |
| min_price | over1 | 股价 > $1 |

**回退候选列表**：FFIE, MULN, ATER, BBIG, RDBX, TBLT, CLOV, GOEV, PROG, ISPC, ESSC, BFRI, IRNT, SPRT, NILE, CEI, FAMI, PIXY。

**准入门控**（在 `analyze()` 中执行）：

| 门控 | 配置参数 | 默认值 |
|------|----------|--------|
| 最低 RVOL | `min_rvol` | 2.0 |
| 价格范围 | `min_price` / `max_price` | $1.00 ~ $50.00 |
| 行业排除 | 硬编码 | Shell Companies, Blank Checks, Closed-End Funds (3类), ETF, SPAC |

行业排除列表（`_EXCLUDED_INDUSTRIES`）过滤以下 NAV 锚定型实体，因为它们的价格行为不符合技术突破模型：

```
Shell Companies, Blank Checks, Closed-End Fund - Equity,
Closed-End Fund - Debt, Closed-End Fund - Foreign,
```


#### 3.4.5 特殊机制

**RVOL 价格方向验证**：v0.4.0 新增。高 RVOL 配合价格下跌不是突破信号而是分配（Distribution），系统会自动打折。这防止了"暴跌放量"被误判为看涨信号。

### 3.5 蓝队 -- 动量催化复合精英队 (MomentumCatalystScreener)

**文件**: `src/teams/blue/screener.py`
**配置键**: `blue_team`

#### 3.5.1 使命与定位

蓝队分析**多时间框架动量与催化剂融合信号**。与其他五支团队专注于特定机制（做空/期权/情绪/突破/结构）不同，蓝队评估的是更广泛的动量健康度、基本面质量和宏观市场环境。

蓝队是唯一一支考虑**市场整体环境（VIX 水平、SPY 趋势）**的团队，在高波动环境下会主动降低评分权重。

> **v1.0.0 重大变更**：蓝队从静态大盘股列表转型为动态 Finviz 小盘动量扫描。AAPL、MSFT、GOOG、NVDA、META、AMZN 等大盘股已被完全移除——它们不可能翻 10 倍。权重从 0.9 降至 0.5（动量单独作为信号较嘈杂，需要与其他团队共振才有价值）。

#### 3.5.2 候选股筛选逻辑 (fetch_candidates)

**主数据源**：Finviz 动态小盘动量扫描（`get_small_cap_momentum_candidates()`）

| 筛选项 | 默认值 | 含义 |
|--------|--------|------|
| 市值上限 | `smallunder2b`（< $2B） | 专注小盘股 |
| 相对成交量 | `over1.5`（> 1.5x） | 有活跃交易信号 |
| 最低价格 | `over1` | 排除仙股 |
| 日均成交量 | `over200k` | 最低流动性要求 |

**回退候选列表**：当 Finviz 失败时使用小/微盘股动量列表（GME, AMC, PLTR, SOFI, CLOV, FUBO, MARA, RIOT 等 18 只标的）。所有大盘股已移除。

**准入门控**：
- `min_momentum_score`（默认 40，v1.0.0 从 60 降至 40）-- 综合动量分数低于此阈值直接跳过。小盘股动量信号更 choppy，门控需更宽容。
- `max_market_cap_millions`（默认 2000）-- 市值超过 $2B 直接跳过。

#### 3.5.3 四因子评分模型 (总分 0-100，受 VIX 乘数影响)

##### 因子一：价格动量 (price_momentum) -- 0~25 分

**综合动量分数计算**：多时间框架加权

| 时间框架 | 权重 | 说明 |
|---------|------|------|
| 5 日 | 15% | 近期动量 |
| 10 日 | 0% | 采集但不计入权重 |
| 21 日 | 25% | 月度趋势 |
| 63 日 | 30% | 季度趋势（最重要） |
| 126 日 | 20% | 半年趋势 |
| 252 日 | 10% | 年度趋势 |

**归一化公式**：每个时间框架的收益率通过 `max(0, min(100, ret + 50))` 映射到 0-100 区间。0% 收益率 = 50 分（中性），-50% = 0 分，+50% = 100 分。对称设计。

**子指标 A -- 综合动量分数 -> 缩放：0~15 分**

`score = momentum_score x 0.15`（满分 100 x 0.15 = 15 分）

**子指标 B -- 趋势一致性：0~10 分**

检查 5 日、21 日、63 日收益率中有多少为正值。

`score = (正值个数 / 总个数) x 10.0`

例如 3 个全为正 = 10.0 分，2 个为正 = 6.7 分。

##### 因子二：催化剂临近 (catalyst_proximity) -- 0~25 分

**子指标 A -- 财报临近度：0~15 分**

| 距下次财报天数 | 分数 | 含义 |
|--------------|------|------|
| 1 ~ 7 天 | 15.0 | 财报即将发布 |
| 8 ~ 14 天 | 10.0 | 预热期 |
| 15 ~ 30 天 | 5.0 | 早期关注 |
| 31 ~ 60 天 | 2.0 | 远期 |
| > 60 天或无数据 | 0.0 | -- |

（`earnings_lookahead_days` 配置为 14，但评分覆盖到 60 天）

**子指标 B -- 分析师覆盖：0~5 分**

| 条件 | 分数 | 含义 |
|------|------|------|
| 有 EPS 预估 | 5.0 | 有分析师跟踪 |
| 无 | 0.0 | 无分析师覆盖 |

注意：营收增长率仅在财务质量因子中评分，不在催化剂因子中重复计数。

##### 因子三：财务质量 (financial_quality) -- 0~25 分

**妖股友好设计**：每只股票基线 5 分。动量策略中，财务质量是加分项而非淘汰项 -- 许多妖股基本面很差但动量极强。

**子指标 A -- 基线分：5 分（所有标的）**

**子指标 B -- 营收增长率 (YoY)：0~8 分**

| 增长率 | 额外分数 |
|--------|----------|
| >= 100% | +8.0 |
| >= 50% | +6.0 |
| >= 10%（`min_revenue_growth_pct`） | +4.0 |
| >= 0% | +2.0 |
| >= -20% | +1.0 |
| < -20% 或无数据 | +0.0 |

**子指标 C -- 资产负债率 (D/E, Debt-to-Equity)：0~5 分**

| D/E 值 | 额外分数 | 含义 |
|--------|----------|------|
| < 0（负数）| +0.0 | 资不抵债，不是"低负债" |

| < 1.0 | +4.0 | 低负债 |
| < 3.0（`max_debt_to_equity`） | +2.0 | 中等 |
| >= 3.0 或无数据 | +0.0 | 高负债或无数据 |

注意：yfinance 返回的 D/E 是百分比形式，代码中除以 100 转换为比率。v0.4.1 修复了负 D/E 被误判为"极低负债"的 bug。

**子指标 D -- 分析师覆盖：0~5 分**（v1.0.0 调整）

| 条件 | 额外分数 | 含义 |
|------|----------|------|
| 有 EPS 预估 | +5.0 | 有分析师跟踪（v1.0.0 从 7 降至 5） |
| 无 EPS 预估 | +2.0 | "隐藏宝石"加分 — 无分析师覆盖意味着雷达外的 10x 潜力 |

理论最高 = 5 + 8 + 5 + 5 = 23 分（有分析师）或 5 + 8 + 5 + 2 = 20 分（无分析师）。理论最低 = 5 分（基线）。

##### 因子四：市场环境 (market_regime) -- 0~25 分

**子指标 A -- VIX 市场状态：0~15 分**

| SPY 5日涨幅 | 分数 |

#### 3.5.4 VIX 全局乘数（重要特殊机制）

蓝队在计算完四因子原始总分后，会应用一个**全局 VIX 折扣乘数**：

| VIX 水平 | 乘数 | 效果 |
|----------|------|------|
| >= 35 | x 0.5 | 红色警报：总分减半 |
| >= 25 | x 0.7 | 黄色警告：总分打 7 折 |
| < 25 | x 1.0 | 正常 |

**设计意图**：在高波动环境下，动量策略的可靠性显著降低。此乘数是一个"安全阀" -- 它影响的是整体评分而非单一因子，确保 VIX 飙升时蓝队评分被有效压制。

**示例**：VIX = 30 时，原始总分 70 分 -> 实际输出 70 x 0.7 = 49 分。

#### 3.5.5 配置参数

```yaml
blue_team:
  lookback_months: 14         # 历史数据回看月数（252 交易日需约 14 月数据）
  max_lookback_months: 18     # 最大回看月数
  momentum_periods:           # 动量计算的时间框架（交易日）
    - 5
    - 10
    - 21
    - 63
    - 126
    - 252
  min_momentum_score: 40      # 综合动量分数门控（v1.0.0 从 60 降至 40）
  max_market_cap_millions: 2000  # 市值上限（v1.0.0 新增）
  earnings_lookahead_days: 14 # 财报关注前瞻天数
  min_earnings_surprise_pct: 5.0 # 最低 EPS 超预期比例（Phase 2）
  vix_high_threshold: 25      # VIX 高阈值
  vix_low_threshold: 15       # VIX 低阈值
  market_breadth_threshold: 0.5 # 市场广度阈值
  min_revenue_growth_pct: 10.0 # 最低营收增长率
  max_debt_to_equity: 3.0     # 最大 D/E 比率
```

#### 3.5.6 信号解读

| 分数区间 | 含义 |
|----------|------|
| 70-100 | 极强动量+催化融合：多时间框架看涨 + 财报临近 + 良好基本面 + 友好市场 |
| 50-70 | 强信号 |
| 30-50 | 中等（可能受 VIX 乘数压制） |
| 10-30 | 弱 |
| 0-10 | 不通过动量门控或市场环境极端不利 |

**常见陷阱**：
- VIX >= 35 时即使基本面和动量很好，蓝队评分也会被砍半
- `min_momentum_score = 60` 门控比较严格，许多盘整中的股票会被直接过滤
- 负 D/E 是资不抵债的信号，不是低负债 -- v0.4.1 已修复此误判
- 动量是滞后指标：所有时间框架都看涨可能意味着趋势已充分反映

### 3.6 紫队 -- 十倍潜力评估师 (TenBaggerScreener)

**文件**: `src/teams/purple/screener.py`
**配置键**: `purple_team`
**版本**: v1.0.0 新增
**综合权重**: 1.3（最高权重）

#### 3.6.1 使命与定位

紫队是 v1.0.0 新增的核心团队，拥有**最高综合评分权重（1.3）**。与其他五支团队专注于特定催化机制不同，紫队评估的是一只股票是否具备**翻 10 倍以上的结构性潜力**。

紫队回答的核心问题是：*"如果所有条件都对齐，这只股票有没有十倍的容量（Capacity）？"*

理想标的画像：微型市值（< $300M）、极小流通盘（< 20M 股）、高内部人持股、低机构持仓、健康现金流（无稀释压力）、波动率正在压缩（蓄势待发）。

#### 3.6.2 候选股筛选逻辑 (fetch_candidates)

**主数据源**: Finviz 免费版筛选器

| 筛选项 | 默认值 | 含义 |
|--------|--------|------|
| 市值上限 | `cap_smallunder`（< $2B） | 只关注小盘/微盘股 |
| 日均成交量 | `sh_avgvol_o200`（> 20 万股） | 最低流动性 |
| 最低价格 | `sh_price_o1`（> $1） | 排除仙股 |

**回退候选列表**：21 只历史上波动剧烈的微/小盘股（FFIE, MULN, ATER, CLOV, GOEV, FUBO, SOFI, MARA, RIOT 等）。

**准入门控**（在 `analyze()` 中执行）：

| 门控条件 | 配置参数 | 默认值 | 说明 |
|----------|----------|--------|------|
| 最大市值上限 | `max_market_cap_millions` | 2000（$2B） | 市值超过此值直接跳过 |
| 最少历史数据 | — | 20 根日线 | 少于 20 根 K 线无法计算技术指标 |

#### 3.6.3 四因子评分模型 (总分 0-100)

##### 因子一：市值层级 (market_cap_tier) -- 0~25 分

**评分依据**：市值越小，翻 10 倍的空间越大。

使用**分段断点映射（Piecewise Breakpoints）**：

| 市值范围 | 分数 | 含义 |
|---------|------|------|
| < $50M | 25.0 | 纳米盘：10x 仅需涨至 $500M |
| $50M ~ $300M | 21.0 | 微盘 |
| $300M ~ $500M | 17.0 | 小微盘 |
| $500M ~ $1B | 13.0 | 小盘 |
| $1B ~ $2B | 8.0 | 小中盘 |
| $2B ~ $5B | 3.0 | 中盘（通过门控的极端情况） |
| >= $5B | 0.0 | 大盘：不可能 10x |
| 数据缺失 | 5.0 | 保守默认分 |

##### 因子二：流通结构 (float_structure) -- 0~25 分

评估流通盘紧度、内部人持股对齐度和机构持股水平。

**子指标 A -- 流通盘大小：0~10 分**

| 流通盘 | 分数 | 含义 |
|--------|------|------|
| < 5M 股 | 10.0 | 极小流通盘，高度易爆 |
| 5M ~ 10M | 8.0 | 很小 |
| 10M ~ 20M | 6.0 | 小 |
| 20M ~ 50M | 4.0 | 中等 |
| 50M ~ 100M | 2.0 | 偏大 |
| >= 100M 或无数据 | 0.0 | 大流通盘 |

**子指标 B -- 内部人持股（Insider Ownership）：0~8 分**

| 内部人持股比例 | 分数 | 含义 |
|---------------|------|------|
| >= 30% | 8.0 | 高度对齐（创始人深度绑定） |
| >= 20% | 6.0 | 良好 |
| >= 10% | 4.0 | 中等 |
| >= 5% | 2.0 | 偏低 |
| < 5% 或无数据 | 0.0 | 无对齐信号 |

**子指标 C -- 机构持股（低 = 更好）：0~7 分**

| 机构持股比例 | 分数 | 含义 |
|-------------|------|------|
| < 20% | 7.0 | 高度散户驱动，挤压潜力最大 |
| 20% ~ 40% | 5.0 | 中等机构参与 |
| 40% ~ 60% | 3.0 | 机构主导中 |
| >= 60% | 1.0 | 高度机构化（有底部支撑但爆发力弱） |
| 无数据 | 0.0 | — |

##### 因子三：稀释风险 (dilution_risk) -- 0~25 分（反向指标：低风险 = 高分）

评估公司是否有稀释股东的动机和能力。

**子指标 A -- 现金健康度（Cash / Debt）：0~10 分**

| 条件 | 分数 | 含义 |
|------|------|------|
| 无债务且有现金 | 10.0 | 堡垒级资产负债表 |
| 现金 > 2× 债务 | 10.0 | 极强现金位 |
| 现金 > 债务 | 7.0 | 健康 |
| 现金 > 50% 债务 | 4.0 | 可控 |
| 现金 < 50% 债务 | 1.0 | 高债务压力 |
| 数据缺失 | 3.0 | 保守默认 |

**子指标 B -- 现金流可持续性（OCF/FCF）：0~8 分**

| 条件 | 分数 | 含义 |
|------|------|------|
| 经营现金流 > 0 | 8.0 | 自我造血，无稀释压力 |
| OCF < 0 但 FCF > 0 | 4.0 | 异常但可控 |
| 现金跑道 > 12 月 | 2.0 | 烧钱但时间充裕 |
| 以上都不满足 | 0.0 | 高稀释风险 |
| 数据缺失 | 2.0 | 保守默认 |

**子指标 C -- 股份结构紧度（Float / Outstanding）：0~7 分**

| 流通/总股本比率 | 分数 | 含义 |
|---------------|------|------|
| < 50% | 7.0 | 大量锁仓，低稀释风险 |
| 50% ~ 80% | 5.0 | 中等锁仓 |
| > 80% | 2.0 | 几乎全流通 |
| 数据缺失 | 3.0 | 保守默认 |

##### 因子四：爆发蓄势 (explosive_setup) -- 0~25 分

检测技术面上的"弹簧压缩"和筹码吸纳模式。

**子指标 A -- 布林带挤压（Volatility Compression）：0~8 分**

计算当前布林带宽度与 20 日平均宽度的比率（ratio = current / avg），比率越低说明波动率压缩越极端。

| 宽度比率 | 分数 | 含义 |
|---------|------|------|
| < 0.5 | 8.0 | 极端压缩（"弹簧"即将释放） |
| 0.5 ~ 0.7 | 5.0 | 显著压缩 |
| 0.7 ~ 0.9 | 3.0 | 轻微压缩 |
| >= 0.9 或数据不足 | 1.0 | 正常或扩张 |

**子指标 B -- 52 周区间位置：0~6 分**

| 在 52 周范围中的位置 | 分数 | 含义 |
|-------------------|------|------|
| 10% ~ 30%（甜蜜点） | 6.0 | 近底部但已反弹，最大上行空间 |
| 30% ~ 50% | 4.0 | 中低位 |
| 80% ~ 100% | 3.0 | 高位可能继续突破 |
| 0% ~ 10% | 2.0 | 仍在下跌，风险较高 |
| 50% ~ 80% | 1.0 | 中高位 |

**子指标 C -- OBV 趋势确认：0~6 分**

检测最近 10 个交易日的 OBV（On-Balance Volume）与价格的关系。

| 条件 | 分数 | 含义 |
|------|------|------|
| OBV 上升 + 价格平/跌 | 6.0 | 看涨背离：隐蔽吸筹 |
| OBV 上升 + 价格上升 | 4.0 | 确认上涨趋势 |
| OBV 持平 | 2.0 | 无明显信号 |
| OBV 下降 | 0.0 | 资金流出 |

**子指标 D -- 近期动量（5 日回报）：0~5 分**

| 5 日涨幅 | 分数 | 含义 |
|---------|------|------|
| 5% ~ 30% | 5.0 | 强势但未过热 |
| 0% ~ 5% | 3.0 | 温和看涨 |
| -10% ~ 0% | 2.0 | 可能反弹 |
| > 30% | 1.0 | 可能过度延伸 |
| < -10% | 0.0 | 深度下跌 |

## 第四部分：综合评分机制

### 4.1 团队权重

综合评分（`composite_score`）是各团队评分的加权平均值。权重配置在 `config/default.yaml` 的 `orchestrator.team_weights` 节中：

| 团队 | 权重 | 说明 |
|------|------|------|
| 红队 (red) | 1.0 | 空头挤压，信号较可靠 |
| 橙队 (orange) | 1.0 | Gamma 挤压，市场微观结构信号 |
| 黄队 (yellow) | **0.8** | 社交情绪较嘈杂，降权处理 |
| 绿队 (green) | 1.0 | 技术突破，信号客观 |
| 蓝队 (blue) | **0.9** | 动量+催化，轻微降权 |

### 4.2 composite_score 计算公式

```
composite_score = (red_score x 1.0 + orange_score x 1.0 + yellow_score x 0.8
                 + green_score x 1.0 + blue_score x 0.9)
                 / (1.0 + 1.0 + 0.8 + 1.0 + 0.9)
```

全部 5 支团队运行时，总权重 = 4.7。

**注意**：未命中（score = 0）的团队**仍然参与分母计算**。这意味着一支团队给出 80 分但其他四支给出 0 分时，综合分数 = 80 x 权重 / 4.7，远低于 80。

### 4.3 team_count 的含义

`team_count` 统计对该标的评分 > 0 的团队数量（0 到 5）。

| team_count | 含义 |
|-----------|------|
| 5 | 全面确认：所有维度都给出信号 |
| 4 | 强交叉验证 |
| 3 | 多维确认 |
| 2 | 双维确认 |
| 1 | 单维信号，置信度较低 |

### 4.4 解读综合排名

**为什么单团队命中也可能排名靠前？**

因为加权平均机制下，一支团队给出极高分（如 95 分 x 1.0 权重 / 4.7 = 20.2 分综合）仍然可能高于另一个标的的多团队中等分（如 5 队各 15 分 x 各权重 / 4.7 = 13.7 分综合）。

**最佳实践**：优先关注 `team_count >= 3` 且 `composite_score >= 40` 的标的 -- 多维交叉确认显著提高信号质量。

### 4.5 权重调整策略

可在 `config/default.yaml` 中修改权重以偏向特定策略：

```yaml
orchestrator:
  team_weights:
    red: 1.5    # 偏向空头挤压
    orange: 1.2 # 重视 Gamma 信号
    yellow: 0.5 # 进一步降低情绪噪音
    green: 1.0
    blue: 0.8
```

权重读取优先级（3 级回退）：
1. 代码中显式传入的 `team_weights` 参数
2. YAML 配置 `orchestrator.team_weights`
3. 硬编码回退值 `_FALLBACK_TEAM_WEIGHTS`

## 第4.5部分：AI 智能重评分

> v0.5.0 新增功能。需要 Anthropic API 密钥和 `anthropic` Python 包。

### 概述

AI 智能重评分是一个**可选的定性评估层**，在五支团队完成量化评分后、综合评分计算前介入。它调用 Claude Opus 4.6 API，让 AI 以团队专家的视角重新审视每个标的的量化评分，捕捉公式无法识别的细微模式。

**核心价值**：

| 问题 | 量化模型的盲区 | AI 可以识别 |
|------|-------------|-----------|
| 拉高出货 (Pump & Dump) | 高 SI + 社交热度 = 高分 | 账号年龄/协调模式异常 |
| 虚假突破 | 放量+价格突破 = 高分 | 分配量伪装为突破量 |
| 回音室效应 | 高提及+高看涨 = 高分 | 单一信息源的同质化放大 |
| IV 碾压风险 | 高 IV = 高分 | 事件后 IV 坍塌的历史规律 |
| 动量耗竭 | 多周期上涨 = 高分 | 过度延伸后的均值回归概率 |

### 工作流

```
5支团队并行运行
    → list[ScreenResult]
    → _build_team_dataframes()
    → [AI 重评分]                    ← 仅当 --ai-rescore + API 可用
       每支活跃团队一次 API 调用
       批量发送该队所有 ticker 的评分和信号数据
       Claude 返回 JSON: ai_score + ai_confidence + ai_reasoning + ai_flags
       混合评分: blended = quant × (1 - ai_weight) + ai × ai_weight
    → _merge_and_score()             ← 使用混合后的分数
    → 输出
```

### 混合评分公式

```
{team}_score = 量化评分 × (1 - ai_weight) + AI评分 × ai_weight
```

默认 `ai_weight = 0.3`，即量化占 70%，AI 占 30%。

**示例**：红队量化评分 = 68，AI 评分 = 72
→ `blended = 68 × 0.7 + 72 × 0.3 = 47.6 + 21.6 = 69.2`

ai_weight 可在 `config/default.yaml` 中调整，范围建议 0.1 ~ 0.5。

### 五支团队的 AI 评估视角

| 团队 | AI 专家角色 | 重点评估维度 | 可能标记的风险/机会 |
|------|-----------|------------|------------------|
| 红队 | 空头挤压专家 | 挤压概率、陷阱风险、机构仓位 | `pump_and_dump_risk`, `squeeze_timing_imminent`, `stale_data_risk`, `institutional_trap` |
| 橙队 | 期权微观结构专家 | Gamma 翻转概率、做市商仓位、IV 碾压风险 | `gamma_flip_imminent`, `iv_crush_risk`, `max_pain_magnet`, `dealer_short_gamma` |
| 黄队 | 社交情绪法医 | 机器人/操纵风险、回音室、有机 vs 人造情绪 | `bot_manipulation_risk`, `echo_chamber`, `sentiment_divergence`, `organic_momentum` |
| 绿队 | 技术突破专家 | 突破可持续性、虚假突破、量能质量 | `false_breakout_risk`, `distribution_volume`, `clean_breakout`, `supply_zone_overhead` |
| 蓝队 | 动量催化分析师 | 动量耗竭、催化剂质量、板块轮动 | `momentum_exhaustion`, `catalyst_approaching`, `catalyst_priced_in`, `regime_mismatch` |

### AI 信心等级

| 等级 | 含义 | 建议权重 |
|------|------|---------|
| `high` | AI 对其评估高度确信，数据充分 | 信赖 AI 评分 |
| `medium` | 数据部分充分或存在模糊信号 | 参考但不完全依赖 |
| `low` | 数据不足或信号矛盾 | 以量化评分为主 |

### 成本估算

使用 Claude Opus 4.6（`claude-opus-4-6`）：

| 项目 | Token 估算 |
|------|-----------|
| 系统提示（每队） | ~500 tokens |
| 用户消息（20 标的） | ~3,500-4,000 tokens |
| AI 响应（20 标的） | ~2,000 tokens |
| **单次调用合计** | ~6,000-7,000 tokens |
| **全扫描（5 队）** | ~30,000-35,000 tokens |

### 缓存

AI 评分结果独立缓存，TTL 默认 2 小时（`ai_rescore.cache_ttl_hours`），短于市场数据缓存（4 小时）。同一组 ticker 的同一团队在 TTL 内第二次调用会直接使用缓存，不消耗 API。

### 优雅降级

AI 重评分遵循"绝不崩溃管线"原则：

| 场景 | 处理方式 |
|------|---------|
| `anthropic` 包未安装 | WARNING 日志，使用纯量化评分 |
| API 密钥缺失 | WARNING 日志，使用纯量化评分 |
| API 认证失败 (401) | ERROR 日志，本次运行禁用 AI，不重试 |
| API 超时/5xx | 指数退避重试 3 次，失败后跳过该团队 |
| API 频率限制 (429) | 指数退避重试 3 次 |
| JSON 解析失败 | 尝试去除 Markdown 代码围栏后重新解析，失败则跳过 |
| AI 仅返回部分 ticker | 缺失的 ticker 使用纯量化评分 |
| AI 评分超出 0-100 | 截断至 [0, 100] 并记录 WARNING |

## 第六部分：缓存与性能

### 6.1 缓存机制

| 缓存类型 | 格式 | 用途 |
|---------|------|------|
| DataFrame 缓存 | Parquet (.parquet) + 元数据 (.meta) | 价格历史、期权链 |
| JSON 缓存 | JSON (.json) | 基本面信息、情绪数据、Google Trends |

**缓存路径**：`data/cache/`（可通过 `general.cache_dir` 配置）

**缓存键**：`{namespace}_{sha256_hash}.{ext}`，其中 hash 基于 `namespace::key`。

**TTL**：默认 4 小时（`general.cache_ttl_hours`）。超过 TTL 的缓存自动失效，下次访问时重新获取。

**原子写入**：所有缓存写入通过 `_atomic_write()` 实现临时文件写入 + `os.replace()` 原子重命名，防止并行线程读到半写文件。

### 6.2 清除缓存

```bash
# 方式一：CLI 参数
python main.py --clear-cache

# 方式二：手动删除
rm -rf data/cache/*
```

`--clear-cache` 会在扫描前删除所有 `.parquet`、`.json`、`.meta`、`.tmp` 文件。

**何时清除缓存**：
- 怀疑数据过时（如市场收盘后数据更新）
- 遇到缓存相关报错
- 需要完全新鲜的数据进行分析

### 6.3 并行 vs 串行

**并行模式（默认）**：
- 使用 `ThreadPoolExecutor`，`max_workers` 默认 4
- 五支团队同时运行，总耗时约等于最慢的团队
- 适合正常使用

**串行模式（--no-parallel）**：
- 团队按顺序依次运行
- 适合调试（日志输出更清晰，错误更容易定位）
- 总耗时 = 所有团队耗时之和

```bash
# 调试时推荐
python main.py --no-parallel --log-level DEBUG
```

### 6.4 性能优化建议

1. **首次运行后利用缓存**：首次扫描最慢（需获取所有数据），后续在 TTL 内大幅加速
2. **减少团队数量**：`--teams red green` 只加载和运行两支团队
3. **减少标的数量**：`--tickers GME AMC` 跳过候选发现直接分析
4. **调整 TTL**：盘后分析可设置 `cache_ttl_hours: 24`
5. **网络超时**：`network_timeout_seconds` 默认 30 秒，网络慢时可适当增大
6. **yfinance 超时限制**：yfinance 不暴露 timeout 参数，其内部超时由操作系统 socket 超时控制（通常 60-120 秒），`network_timeout_seconds` 仅适用于 Finviz 等直接 HTTP 请求
7. **Google Trends 限速**：被限速后熔断器自动开启（3 次连续失败后跳过），无需手动干预

### 7.5 常见问题排查

#### API 相关

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| Finviz 返回空结果 | 频率限制（免费版） | 等待几分钟后重试，或使用 `--clear-cache` |
| yfinance 挂起无响应 | Yahoo Finance 服务器慢或限流 | 等待超时（60-120s），或用 `--no-parallel` 减少并发 |
| Reddit 数据全为空 | 未配置 API 凭据 | 参见 5.2 节配置 secrets.yaml |
| Google Trends 全为 None | 被 Google 限速 | 熔断器自动开启（正常行为），减少标的数量或等待 |
| ApeWisdom 请求失败 | API 不可用 | 黄队自动降级到回退列表 |

#### 评分相关

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| 蓝队评分突然变低 | VIX 升高触发全局乘数 | 检查 metadata 中的 `regime_multiplier` 值 |
| 黄队情绪极性始终为 0 | 无 Reddit 凭据 | 配置 secrets.yaml 或设置环境变量 |
| 红队 SI 数据标记为"过期" | FINRA 报告延迟 | 正常现象，注意日志中的 WARNING |
| 绿队放量但得分低 | 价格下跌触发方向折扣 | 检查 log 中的 "discounting volume score" 信息 |
| 所有团队返回 0 结果 | 市场休市或网络问题 | 检查日志确认数据获取状态 |

#### AI 重评分相关

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| "anthropic package not installed" | 未安装 anthropic 包 | `pip install anthropic` 或 `pip install -e ".[ai]"` |
| "No Anthropic API key configured" | API 密钥未设置 | 设置 `ANTHROPIC_API_KEY` 环境变量或配置 secrets.yaml |
| "AI re-scoring disabled due to authentication failure" | API 密钥无效 | 检查密钥是否正确、是否过期 |
| AI 评分全部与量化一致 | 缓存命中 | `--clear-cache` 清除 AI 缓存后重试 |
| `--ai-rescore` 无任何效果 | SDK 未安装或密钥缺失 | 检查日志中的 WARNING 信息 |
| AI 响应解析失败 | Claude 返回非 JSON 格式 | 系统自动降级为量化评分，检查 ERROR 日志 |

#### 配置相关

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| YAML 解析错误 | YAML 语法错误 | 检查缩进和冒号，v0.4.1 会提示具体出错文件 |
| 配置修改不生效 | 缓存中的旧数据 | `--clear-cache` 清除后重试 |
| 环境变量不生效 | 变量名拼写错误 | 确认使用大写：`REDDIT_CLIENT_ID` |
